## 版本管理

### 版本号管理

项目版本号统一在 [app/config.toml](../app/config.toml) 的 `[project]` 部分管理（Single Source of Truth）：

```toml
[project]
name = "bensz-channel"
version = "1.42.1"
description = "现代化 Web 社区平台，基于 PHP、PostgreSQL、Docker 构建，适合团队协作与知识分享"
```

### 创建新版本

1. **更新版本号**：修改 [app/config.toml](../app/config.toml) 中 `[project]` 部分的 `version` 字段
2. **更新变更日志**：在 [CHANGELOG.md](../CHANGELOG.md) 中添加新版本的变更内容
3. **创建 Release**：在 GitHub Actions 页面手动触发 `Create Release from config.toml` 工作流
   - 工作流会自动从 app/config.toml 读取版本号
   - 自动创建 Git tag（格式：`v{version}`）
   - 自动从 CHANGELOG.md 提取 Release Notes
   - 创建 GitHub Release

### 自动发布 Docker 镜像

项目包含 GitHub Actions 工作流 `publish-release-images`，会按 `0 18 * * *`（UTC 每天 18:00，即北京时间 02:00）自动检查最新的 **已发布 GitHub Release**。

当 Docker Hub 中还不存在该 Release 对应的镜像标签时，工作流会自动构建并推送：

- `DOCKERHUB_NAMESPACE/<仓库名>-web:<release-tag>` 与 `:latest`
- `DOCKERHUB_NAMESPACE/<仓库名>-auth:<release-tag>` 与 `:latest`

**配置要求**：

Repository Variables：
- `DOCKERHUB_NAMESPACE`：Docker Hub 命名空间，必填（若未单独指定镜像仓库）
- `DOCKERHUB_WEB_REPOSITORY`：可选，覆盖默认的 Web 镜像仓库名
- `DOCKERHUB_AUTH_REPOSITORY`：可选，覆盖默认的 Auth 镜像仓库名

Repository Secrets：
- `DOCKERHUB_USERNAME`：Docker Hub 用户名
- `DOCKERHUB_TOKEN`：Docker Hub Access Token

**构建使用的项目资源**：

Web 镜像（[docker/web/Dockerfile](../docker/web/Dockerfile)）：
- [app/](../app/) - Laravel 应用主目录（PHP 后端 + 前端资源）
- [config/](../config/) - 项目配置文件
- [docker/web/](../docker/web/) - Web 容器配置（entrypoint、supervisord）
- [docker/nginx/](../docker/nginx/) - Nginx 服务器配置

Auth 镜像（[auth-service/Dockerfile](../auth-service/Dockerfile)）：
- [auth-service/](../auth-service/) - 认证服务（Node.js）
- [config/](../config/) - 项目配置文件（与 Web 共享）

### 版本同步检查

项目包含自动版本同步检查工作流，会在以下情况触发：
- 修改 app/config.toml 或 CHANGELOG.md 时
- 创建 Pull Request 时

检查内容：
- app/config.toml 版本号与最新 GitHub Release 是否同步
- CHANGELOG.md 是否包含当前版本的变更记录
- 自动在 PR 中添加版本状态评论

## 数据持久化

- PostgreSQL 数据显式挂载到 `./data/postgres`
- Redis AOF 显式挂载到 `./data/redis`
- Mailpit 数据库显式挂载到 `./data/mailpit`
- Laravel 运行时存储与静态页面输出显式挂载到 `./data/web/`
- 文章与评论里通过粘贴上传的图片与视频统一存放在 `./data/web/storage/app/public/media/`，重部署后仍会继续保留，备份整个 `./data/` 目录即可连同数据库和媒体资源一起保存
- 用户上传的自定义头像统一存放在 `./data/web/storage/app/public/avatars/`，支持 `JPG/PNG` 且单文件限制 `1MB`，重部署后同样会随持久化目录保留
- Markdown 视频粘贴上传默认允许到 `500MB`，可通过 `VIDEO_UPLOAD_MAX_MB` 调整；Docker 内置的 Nginx/PHP 上传限制已同步提升到可承接该能力
- Web 容器启动时只执行 `migrate + SystemBootstrapSeeder`，避免重部署时反复写入 demo 内容
- 仓库更新时只需执行 `git pull` 后依次运行 `./scripts/build.sh` 与 `./scripts/compose.sh up -d`

## 认证架构

### 认证系统概述

- Laravel 继续负责页面渲染、业务权限与本地 Session
- `auth-service/` 负责 OTP 发送与验证，底层由 Better Auth 自托管管理
- 邮箱验证码通过 Mailpit/SMTP 投递；手机号验证码保留演示模式回调
- 微信 / QQ 扫码登录由 Laravel 直接对接各自官方 OAuth，避免把第三方网页登录协议强塞进 Better Auth OTP 服务
- Better Auth 独立使用 PostgreSQL `auth` schema，避免与 Laravel 的 `public` schema 冲突
- Laravel 通过内部共享密钥调用 `auth-service` 的 `/internal/otp/send` 与 `/internal/otp/verify`
- `app/` 与 `auth-service/` 都会在启动早期自动加载 `app/config.toml` 与 `config/.env`

### 用户角色

**管理员**
- 默认预置 1 位管理员账号
- 能力：管理频道、发布文章、管理普通用户资料与角色、删除普通用户、发布评论

**登录成员**
- 邮箱 + 验证码登录
- 邮箱 + 密码登录
- 微信扫码登录（默认演示模式，可切换真实微信开放平台 OAuth）
- QQ 扫码登录（默认演示模式，可切换真实 QQ 互联 OAuth）
- Better Auth 手机验证码链路（后端兼容保留）
- 能力：发表评论

**游客**
- 仅可浏览压缩静态页面
- 不可发文、不可评论

### 微信 / QQ 扫码登录配置

- 项目现在已经同时支持两种运行形态：
  - **演示模式**：默认开启，无需任何外部平台账号，适合 Docker 重部署后直接审查
  - **真实 OAuth 模式**：在微信开放平台 / QQ 互联完成网站应用创建、审核和回调配置后即可启用
- 默认配置下，`app/config.toml` 里的 `WECHAT_QR_MODE` 与 `QQ_QR_MODE` 都是 `demo`，因此完成 `./scripts/build.sh` 与 `./scripts/compose.sh up -d` 后登录页可直接使用演示二维码流程
- 切换到真实模式时：
  - 在 `app/config.toml` 设置 `WECHAT_QR_MODE="oauth"`、`QQ_QR_MODE="oauth"`
  - 填写 `WECHAT_CLIENT_ID`、`QQ_CLIENT_ID`
  - 在 `config/.env` 填写 `WECHAT_CLIENT_SECRET`、`QQ_CLIENT_SECRET`
  - 如需自定义回调地址，再补 `WECHAT_REDIRECT_URI`、`QQ_REDIRECT_URI`
- 若不显式设置回调地址，项目默认使用：
  - 微信：`{APP_URL}/auth/social/wechat/callback`
  - QQ：`{APP_URL}/auth/social/qq/callback`
- 更详细的新手教程见 [如何让本项目支持微信和QQ扫码登陆.md](如何让本项目支持微信和QQ扫码登陆.md)

## 静态页面与 CDN

### 静态 HTML 生成

- 游客 GET 请求优先命中 `public/static/` 内的构建结果
- 每次管理员改文章/频道、成员发评论后，都会触发静态站点重建；高频文章/评论更新默认走异步队列与增量重建
- 静态 HTML 会额外生成 `.gz` 文件，Nginx 开启 `gzip_static`
- 构建产物除首页、频道页、文章页外，还会同步生成 `static/robots.txt` 与 `static/sitemap.xml`
- 首页、频道页、文章页在动态渲染与静态构建两条链路上都会统一输出 canonical、Open Graph、Twitter Card 与 JSON-LD 结构化数据
- 当异步静态重建已排队但新快照尚未落盘时，`storage/app/static-build.pending` 标记会临时关闭游客静态直出，Nginx 会自动回源 Laravel 动态页，确保访客始终优先看到最新内容
- 资源文件采用 Vite 指纹命名，可通过 `ASSET_URL` 接入 CDN

### 搜索引擎发现入口

- `GET /robots.txt`：返回允许抓取公开内容的 robots 规则，并声明 `Sitemap: {APP_URL}/sitemap.xml`
- `GET /sitemap.xml`：自动列出首页、公开频道、公开文章、全站 RSS 与频道 RSS 的绝对地址
- 登录页、后台页、账户设置等非公开 HTML 页面默认输出 `meta name="robots" content="noindex, nofollow"`，避免被搜索引擎误收录
- Nginx 会优先尝试直接返回 `public/static/robots.txt` 与 `public/static/sitemap.xml`；当静态文件尚未生成时，再回退到 Laravel 动态路由

### 静态资源 CDN 配置

当前实现支持两种模式：

| 模式 | 说明 | 适用场景 |
|------|------|----------|
| 回源型 CDN | 静态资源仍保存在 Web 服务，CDN 节点按需回源拉取 | Cloudflare、DogeCloud CDN、独立静态资源域名 |
| 对象存储型 CDN | 将 `public/` 下的公开资源同步到对象存储，再通过 CDN 域名对外提供访问 | 国内加速、减轻源站带宽压力、跨节点分发 |

推荐入口：后台 `CDN 设置` 页面。

页面能力：

- 切换 `回源型 CDN` / `对象存储型 CDN`
- 保存 CDN 草稿配置，不会立刻切换线上运行状态
- 手动点击 `应用 CDN` / `停止 CDN` 控制是否启用当前草稿
- 配置对象存储服务商、桶、区域、端点、Access Key、Secret Key
- 测试连接、查看差异、立即同步、清空远程文件
- 查看最近 50 次工作日志，包含保存、测试、应用、停止、构建、同步、清理及错误细节

#### 回源型 CDN

回源型模式下无需上传文件，只需要设置公开资源域名：

```env
CDN_MODE=origin
CDN_ASSET_URL=https://cdn.example.com
```

效果：

- 页面链接仍使用 `APP_URL`
- `asset()` 生成的 CSS / JS / 图片 URL 自动切换为 `CDN_ASSET_URL`
- 保存后不会立即生效，需要在后台点击 `应用 CDN`

#### 对象存储型 CDN

对象存储模式会把默认公开目录同步到兼容 S3 的对象存储：

- `public/build`
- `public/storage`
- `public/images`
- `public/fonts`

推荐配置：

```env
CDN_MODE=storage
CDN_ASSET_URL=https://assets.example.com
CDN_STORAGE_PROVIDER=dogecloud
CDN_STORAGE_ACCESS_KEY=
CDN_STORAGE_SECRET_KEY=
CDN_STORAGE_BUCKET=bensz-assets
CDN_STORAGE_REGION=auto
CDN_STORAGE_ENDPOINT=https://oss.example.com
CDN_STORAGE_PUBLIC_URL=https://assets.example.com
CDN_SYNC_ENABLED=true
CDN_SYNC_ON_BUILD=true
```

说明：

- `CDN_ASSET_URL` / `CDN_STORAGE_PUBLIC_URL` 用于最终对外访问的资源域名
- `CDN_STORAGE_ENDPOINT` 用于管理对象存储 API
- `CDN_SYNC_ON_BUILD=true` 时，每次静态页构建完成后都会派发 `cdn:sync` 任务
- 后台保存对象存储草稿后，建议先执行 `测试连接`，确认无误后再点击 `应用 CDN`

#### 手动同步命令

```bash
cd app
php artisan cdn:sync
php artisan cdn:sync --queue
php artisan cdn:sync --clear
```

#### 验证方法

```bash
# 检查首页资源链接是否已经切换到公开资源域名
curl -s https://community.example.com | grep -o 'https://[^" ]*build/assets[^" ]*' | head -3
```

- 回源型 CDN：应看到你配置的 CDN 域名
- 对象存储型 CDN：应看到对象存储 CDN 域名，且后台工作日志中的测试/应用/同步状态为 `success`

更详细的步骤见 `docs/CDN配置指南.md`。

- `auth-service/node_modules` 通过符号链接指向 `/Volumes/2T01/Test/bensz-channel/auth-service/node_modules`

- `npm install` 完成后会自动执行 `app/scripts/ensure-managed-node-modules.sh`，把可能被 npm 重建到项目内的依赖目录重新迁回托管路径

## 测试规范

为了让 AI 或人工在改动后快速判断"系统是否仍然正常、稳定、高效工作"，仓库现在提供统一测试入口 `scripts/test/`：

```bash
# 完整回归 + Docker 重部署 + 冒烟 + 稳定性 + 性能
./scripts/test/all.sh

# 按需执行单项验证
./scripts/test/auth-regression.sh
./scripts/test/app-regression.sh
./scripts/test/docker-redeploy.sh
./scripts/test/docker-smoke.sh
./scripts/test/stability.sh
./scripts/test/performance.sh
```

判定口径统一如下：

- `NORMAL`：现有单元/功能回归、前端构建与 Docker 冒烟均通过
- `STABLE`：关键健康检查与管理员登录链路连续多轮通过
- `EFFICIENT`：首页、登录页、RSS 在默认性能预算内
- `SAFE_CHANGE`：以上三项全部通过，可作为"本次改动暂无回归迹象"的自动化证据

## 订阅系统

### SMTP 邮件订阅

- 仅注册用户可用
- 默认开启"全部版块新文章提醒"、"评论 @ 提醒"和"评论回复提醒"
- 登录后可在 `订阅设置` 页面关闭全部提醒，或仅保留指定版块 / 指定标签
- 登录后还可在文章评论区按单条评论线程暂停 / 恢复后续提醒，适合高热长讨论场景
- 登录后也可在 `账户设置` 页面维护邮箱、手机号与密码登录信息
- 管理员可在同一页面测试当前 SMTP 表单配置；测试不会保存当前 SMTP 草稿配置，邮件会发到"测试收件邮箱"字段指定的地址；该测试收件邮箱会在点击"保存 SMTP 配置"后保留，便于下次继续验证
- SMTP 保存后的配置除了文章订阅、评论 @ 提醒、评论回复提醒和邮箱验证码，也会用于管理员活动提醒
- 当有新用户首次注册，或任意已注册用户发布评论时，系统会向 `ADMIN_EMAIL` 发送管理员通知邮件

### RSS 订阅

- 全部版块：`/feeds/articles.xml`
- 单个版块：`/feeds/channels/{channel-public-id}.xml`
- 单个标签：`/feeds/tags/{tag-public-id}.xml`
- 公开文章、频道、标签和对应 RSS 链接统一使用固定长度的不可变 `public_id`，避免标题、slug、频道名或标签名调整后链接失效
- 任何拿到链接的用户都可直接在 RSS 阅读器中订阅

## 管理功能详解

### 用户管理

- 默认管理员邮箱：`admin@example.com`，密码：`admin123456`，用户ID：`0`
- 管理员可在"用户管理"页手动创建新用户，直接设置昵称、邮箱、手机号、头像、简介、角色与初始密码
- 若站点关闭邮箱验证码登录，系统会强制要求填写初始密码，避免创建出无法登录的账号
- 管理员可维护普通用户的昵称、邮箱、手机号、头像链接、简介与角色
- 用户卡片支持折叠 / 展开，右上角统一提供保存、删除、展开 / 收起图标按钮
- 管理员可查看最近 7 天登录 / 活跃、评论与发文仪表盘
- 支持当前页多选批量删除普通用户；删除时会一并清理被删账号的登录会话、密码重置令牌以及关联内容
- 管理员可对普通用户执行 1 天、3 天、7 天、30 天、永久或自定义截止时间的封禁，并可随时解除封禁
- 已封禁账号会被阻止继续登录，以及访问后台和其他需要鉴权的站内功能
- 新注册或首次登录创建的用户会自动获得稳定用户 ID，普通用户从 `101` 开始递增
- 后续修改昵称、邮箱、手机号都不会改变该 ID
- 管理员还可在"用户管理"页使用稳定用户 ID 检索用户

### 频道管理

- 管理员可在"频道管理"页决定哪些频道出现在顶栏导航中
- 频道被隐藏后仍保留访问地址与内容
- `未分类` 默认隐藏但可按需显示
- `精华` 为内置系统频道且默认展示

### 标签管理

- 管理员可在"标签管理"页创建、编辑、删除标签
- 标签只能由管理员定义，普通用户不能新增或修改标签
- 单篇文章可绑定多个标签，标签删除后会自动清理文章绑定关系与标签订阅关系
- 标签管理页会显示相关文章数、标签订阅数，并提供对应 RSS 链接复制入口

### 文章管理

- 管理员可在"文章管理"页一键将文章设为"置顶"或"精华"，也可随时取消
- 管理员可在文章发布 / 编辑表单中为文章勾选多个标签
- 首页会优先展示置顶文章
- "精华"频道会聚合所有精华文章
- 管理后台的大部分高频操作已改为紧凑图标按钮，鼠标悬停会显示用途提示

### 评论管理

- 管理员可在"评论管理"页按评论内容、用户昵称、文章标题和显示状态筛选评论
- 每条评论都可直接查看所属文章，并可在当前页直接回复；回复成功后会回到评论管理页，无需手动跳回文章评论区
- 管理员可管理任意用户发布的评论（包括管理员自己的评论），并一键切换"显示 / 隐藏"或执行删除
- 评论的隐藏、恢复、删除会同步刷新文章公开评论数，并触发对应静态页面增量重建
- 前台评论区已支持评论回复树；任意可见评论都可继续被回复，形成分层讨论
- 普通用户仅可管理自己发布的评论，以及对自己评论继续展开的后续回复；与自己无关的评论不会显示删除入口，后端也会拒绝越权删除
- 评论作者默认会订阅自己发布的评论线程；回复者也会自动订阅自己参与的那条讨论

### 站点设置

- 管理员可在"站点设置"页面覆盖 `app/config.toml` 中 `APP_NAME`、`SITE_NAME`、`SITE_TAGLINE` 的默认值
- 管理员可在"站点设置"页面选择项目时区，默认使用 `Asia/Shanghai`；保存后会统一影响文章发布时间自动填充、页面时间格式、相对时间、RSS、SEO 与备份时间戳
- 管理员可在"CDN 设置"页面维护公开资源域名、对象存储凭证与同步策略；"站点设置"页不再保存 CDN 字段
- 管理员可控制是否开放邮箱验证码、邮箱密码、微信扫码、QQ 扫码四种登录/注册入口
- 管理员可调整文章编辑器图片上传大小上限（默认 50MB）
- Markdown 编辑器支持粘贴图片与不大于 500MB 的视频，视频上限通过 `VIDEO_UPLOAD_MAX_MB` 环境变量控制

### 订阅设置

- 管理员登录后可进入"订阅设置"，在原有订阅偏好之外维护全站 SMTP 服务器、端口、账号、密钥与发件人信息
- 普通成员可在同一页面分别维护按版块订阅和按标签订阅，并直接复制对应 RSS 链接
- 可直接点击"测试 SMTP"按钮验证当前表单配置是否可正常投递
- 保存后的配置会同时用于文章订阅通知、评论提醒、管理员活动提醒与邮箱验证码发送

### 账户设置

- 任意已登录用户都可进入"账户设置"页面
- 自助修改昵称、邮箱、手机号、个人简介与登录密码
- 头像支持三种来源：站内默认风格、外链头像、自定义上传头像
- 默认头像现提供多套可切换风格，包含经典首字母、极光环、轨道脉冲、像素拼图、剪纸层叠
- 自定义上传头像仅接受 `JPG/PNG`，大小不超过 `1MB`
